# アクセシビリティ（WCAG 2.1 AAA）

Lily UI は、これで作ったアプリが WCAG 2.1 AAA に届くように設計しています。アクセシビリティは分担して実現するものです。ライブラリはアクセシブルな部品とガードレールを用意します。最終的な達成は、利用者側の実装と内容の選び方しだいです。

## ライブラリが保証すること

- キーボードフォーカスが見える（2.4.7）。すべての操作可能な部品に一貫した `:focus-visible` のリングを表示します。代わりを用意せずに消すことはありません。
- 動きを抑える（2.3.3）。アニメーションは `prefers-reduced-motion` を尊重します。
- 色だけに頼らない（1.4.1）。状態は色だけでなく、文字やアイコンでも伝えます。
- 文字を拡大できる（1.4.4）。サイズはすべて `rem` を基準にしており、ルートのサイズはユーザーのブラウザ設定に従います。
- コントラスト（1.4.3 / 1.4.6）。標準テーマの文字と背景は全組み合わせで AAA（本文 7:1、大きい文字 4.5:1、非テキスト 3:1）。本文サイズの有色テキスト・リンクには、AAA を満たす `--lily-color-*-text`（`primary-text` / `accent-text` / `danger-text` / `warning-text` / `success-text` / `info-text`）を使います。ソリッドな操作色・状態色の面（`primary` / `danger` / `success` / `info`）は色を深めてあり、面上の文字（明色テーマは白、暗色テーマはほぼ黒）が既定状態で 7:1 以上 — つまり**塗りつぶしコントロールも AAA** です。`warning` は黄色＋ほぼ黒文字で 8.7:1（hover は一時状態で約 6.3:1）。`accent`（藤）は装飾・強調用に鮮やかさを保ちます（コンポーネントが本文を載せる用途はありません）。
- スクリーンリーダー向けのユーティリティとして `.lily-visually-hidden` と `.lily-skip-link` を用意しています。
- 部品の role と状態が正しい。たとえば読み込み中のボタンには `aria-busy`、ネイティブでない無効状態のコントロールには `aria-disabled` を付けます。
- 自動チェックがある。すべての部品に `*.a11y.test.tsx`（vitest-axe）があり、Storybook の a11y アドオンでも検証します。違反があれば CI が失敗します。

## 利用者側で行うこと

- 独自テーマのコントラスト（1.4.3 / 1.4.6 / 1.4.11）。色トークンを上書きしたら、コントラストを確かめ直してください。本文は 7 対 1 以上、大きい文字は 4.5 対 1 以上、UI の境界線は 3 対 1 以上です。
- 内容やアイコンに、意味の伝わるラベルや代替テキストを付けること。
- ページ全体で見出しの構造を筋の通ったものにし、ランドマーク領域を置くこと。
- ナビゲーションやルートの切り替えにあわせてフォーカスを管理すること。

## 「完全な AAA 準拠」について

どんなコンポーネントライブラリも、ページ全体の AAA 準拠を単独で保証することはできません。基準のなかには内容や組み立てに左右されるものがあるからです。選んだ色のコントラスト、見出しの順序、ページのランドマークなどがそれにあたります。Lily UI がめざすのは、アクセシブルな道を初期状態で一番簡単に通れるようにすることと、部品が後退したときにテストや CI ではっきり知らせることです。

## テスト

```bash
pnpm test:a11y     # vitest-axe across components
pnpm dev           # Storybook → a11y panel per story
```
